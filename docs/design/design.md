# 知识大脑系统设计

本文档描述如何把 `think.md` 中的总体思想转化为系统设计。

系统的目标不是构建一个静态知识库，而是构建一个能够吸收知识、激活记忆、组织问题、推理、追踪和学习的个人知识大脑。

知识大脑不是通用 Agent 平台，不负责替代 Agent 平台完成任务执行、环境操作和通用工具编排。

知识大脑的目标，是为用户和 Agent 平台提供底层的知识理解、知识使用和知识沉淀能力。

知识大脑也不是知识图谱系统。知识图谱可以作为表达概念、实体和关系的一种方式，但知识大脑的目标不是构建一张完整知识图，而是让知识能够被理解、激活、组织、验证、追踪和沉淀。

整体设计可以概括为：

```text
材料进入
  -> 形成知识单元和知识点
  -> 使用中形成或验证 ActivationLink
  -> 问题触发思维模式选择
  -> ActivationLink 激活知识
  -> KPN 在必要时补充上下文
  -> Working Model 在复杂问题中组织思考
  -> 生成回答

如果本次问答产生学习价值：
  -> 形成 Learning Event
  -> Study 根据 Learning Event 调整长期记忆
  -> 稳定结果可进入 Wiki 编译
```

知识大脑不是所有问题都走完整推理链路。简单、熟悉、低风险的问题可以轻量处理；复杂设计、判断和决策问题才需要临时认知模型；时效性强、风险高或存在证据冲突的问题，应进入查证或冲突检测。Wiki 页面是长期沉淀结果，不是导入时一次性生成的产物。

## 1. 系统核心对象

### 原始材料

原始材料是外部知识进入系统时的形态。

它可以是文档、笔记、对话、网页、代码、实践记录或其他信息来源。

原始材料不是最终知识，只是知识大脑的输入。

### 知识单元

知识单元是长期记忆的基础编码单位。

它不是为了检索而切出来的文本片段，而是一个相对完整、可理解、可追溯、可复用的知识表达。

一个好的知识单元应该能回答：

```text
它表达了什么知识；
它来自哪里；
它适合在什么问题中被使用；
它能否回到原始材料取证。
```

### 概念

概念不直接承载知识内容。

概念是认知分类、导航入口和问题理解框架。它的作用是帮助系统理解「当前问题属于哪个认知入口」，而不是直接提供可作答的知识条文。

领域和概念是通用认知定义，提供稳定的分类入口和适用边界；它们不承载视角差异化的激活方式。

### KnowledgePoint

KnowledgePoint 是从 KnowledgeUnit 中抽取或沉淀出的可复用知识表达。

KnowledgePoint 才是可被使用、组合、验证和沉淀的知识内容。它的作用是提供「当前问题可使用的知识内容」，而不是充当认知分类。

KnowledgePoint 不是概念，概念也不是 KnowledgePoint 的别名。概念决定从哪个认知入口进入；KnowledgePoint 提供进入后可供调用的知识表达。

### KnowledgePoint Network

KnowledgePoint Network 是知识点之间的轻量语义连接网络。

它连接的不是文档，也不是主题，而是已经从知识单元中提炼出的 KnowledgePoint。

KPN 的详细设计见 `docs/design/kpn.md`。

KPN 的作用不是检索答案，也不是构建知识图谱。

它的作用是在核心 KnowledgePoint 被 ActivationLink 激活之后，补充 Working Model 所需的上下文，帮助组织变量、边界、反例和知识缺口。

KPN 不应独立扩散：只在核心知识点已被激活、且当前问题需要上下文补充时才启用。KPN 不应替代 ActivationLink，也不应成为新的主题聚类层。

整体定位是：

```text
ActivationLink
    负责找到知识

KnowledgePoint Network
    负责补充上下文

Working Model
    负责组织思考

Learning Event / Trace
    负责记录对长期记忆学习有价值的事件和结果

Study
    负责根据 Learning Event 修正长期记忆
```

### 长期记忆

长期记忆是系统已经吸收和组织过的知识。

它由知识单元、概念、知识点、关系、经验、方法、实践路径和证据来源共同构成。

长期记忆不是一次性完成的结构，而是在使用中逐渐稳定。

其中，知识单元和知识点更偏材料侧和证据侧。

领域和概念更偏使用侧和认知侧，是通用认知定义。

认知视角的差异不体现在领域和概念的定义上，而体现在 ActivationLink 上：同一领域和概念下，不同视角可以形成不同的激活路径和使用条件。

### 认知视角

认知视角描述知识大脑当前如何使用知识。

它可以来自：

```text
使用者；
角色；
任务类型；
专业方向；
长期使用环境。
```

例如：

```text
采购管理；
软件架构；
售前投标；
项目管理；
个人学习。
```

### Domain / Concept 与 ActivationLink 的职责边界

领域和概念是通用定义，不承载视角差异化的使用方式：

```text
名称、描述、适用边界等描述稳定的认知入口和分类框架；
回答「这个知识点可能属于哪个认知分类」；
不表达「在某个视角、场景、目标下如何激活某条知识点」。
```

认知视角的差异只体现在 ActivationLink 上：

```text
同一领域和概念下，不同认知视角可以有不同的激活路径；
场景、目标、思维模式、注意焦点等使用条件属于 ActivationLink；
同一知识单元在不同视角下可以有不同的 ActivationLink，或仅有部分视角建立了激活路径。
```

因此，领域和概念应在当前认知视角下被查询和使用，但其语义本身是共享的；视角差异通过 ActivationLink 沉淀。

### 知识结构

知识结构描述长期记忆内部如何组织。

它包括：

```text
领域；
概念；
知识点；
关系；
方法；
经验；
实践路径；
待验证假设。
```

知识结构的作用，是让系统面对问题时能够激活相关记忆，而不是从文本中盲目搜索。

知识结构可以包含图状关系，但不等同于知识图谱。

它服务于知识使用，而不是为了构建一张完整静态图。

第一版中的关系主要承担证据、溯源、索引、激活记录和使用记录，不承担开放知识图谱推理。

### 思维模式

思维模式描述系统面对当前问题时应该采用哪种处理方式。

不同问题不应该都走同一条深度推理链路。

系统需要根据问题类型、知识熟悉度、不确定性、风险和用户意图，选择直接记忆、快速检索、经验路径、工作模型、查证、冲突检测或反思学习等模式。

思维模式不是单个模型提示词，也不是某个检索函数。

它应该落地为一类可以被系统选择和执行的问题处理过程。

在工程实现中，思维模式由 agent 承载。

Agent 负责按照某种思维模式编排多个认知服务，完成一次问题处理。

### 认知能力

认知能力描述知识大脑为了理解、检索、组织、推理、追踪和学习而具备的稳定能力。

例如：

```text
问题理解；
认知结构激活；
材料补充查找；
检索结果融合；
缺口识别；
证据回溯；
临时认知建模；
推理组织；
学习事件记录；
学习沉淀。
```

认知能力不应该直接等同于模型可见的 skill。

它更像系统内部稳定服务。

在工程实现中，认知能力应落地为 service。

Service 负责稳定的数据访问、索引查询、模型调用、错误处理、状态记录和 trace 输出。

模型提示词只参与 service 内部需要语义判断的局部步骤。

因此：

```text
认知能力 -> service；
思维模式 -> agent；
agent 编排 service；
service 在必要时调用模型提示词。
```

### 临时认知模型

临时认知模型是系统围绕本次问题建立的工作模型。

它不是长期记忆，也不是最终回答。

它负责把本次问题相关的概念、知识、证据、缺口、冲突和推理路径组织起来，让系统可以在此基础上思考。

### Learning Event / Trace

Learning Event 是问题处理后产生的学习事件。

Trace 是 Learning Event 的记录形式，详见 `docs/design/trace.md`。

它只记录对长期记忆有价值的事件和结果，不记录完整思考过程。例如：激活路径是否成功、是否存在认知缺口、用户是否纠正、是否发现知识冲突、是否形成候选 ActivationLink、是否需要调整 Concept 或 Wiki、是否需要重新验证某个 KnowledgePoint。

边界说明：

```text
Trace 不是每次问题处理的必经步骤；
Trace 不记录完整思考过程；
Trace 不保存模型推理链；
Trace 只记录学习相关事件；
Study 服务于长期记忆优化，尤其是 ActivationLink；
复盘和反思能力属于 Agent Runtime，不属于 Knowledge Brain 的核心职责。
```

### 学习结果

学习结果是 Study 根据 Learning Event 对长期记忆做出的调整。

它可以强化一个知识点，修正一条关系，补充一个概念，沉淀一条实践路径，或者标记一个知识缺口。

学习结果分属材料层、认知层和表达层。材料层调整知识单元和证据可靠性；认知层调整领域、概念和 ActivationLink；表达层判断哪些主题已足够稳定，适合编译或更新 Wiki 页面。

### Wiki 页面

Wiki 页面是知识大脑在长期使用中逐渐稳定出的主题性知识表达。

它不是原始材料，也不是导入时一次性生成的目录摘要，而是材料层和认知层经过反复激活、验证和反馈后，在表达层编译出的可读、可维护、可演化的长期沉淀。

同一批知识单元在不同认知视角下，可以形成不同的 Wiki 页面。

## 2. 知识如何进入长期记忆

外部材料进入系统后，不应直接成为可使用知识。

它需要先被转换为规范化 Markdown，并保留来源上下文，然后再进入从材料到长期记忆的过程。

### 保留来源

系统首先要保留材料来源。

知识大脑不是凭空记忆，它必须知道每个知识来自哪里。

来源保留的目的不是工程追踪，而是为了让未来回答可以回到证据，并让旧知识在材料变化后可以重新判断。

### 形成知识单元

系统需要把原始材料转换为知识单元。

知识单元的目标不是越短越好，而是表达一个完整的认知动作。

例如：

```text
一个定义；
一个判断；
一个原则；
一个约束；
一种方法；
一个案例；
一个经验。
```

知识单元是长期记忆的最小可用表达。

### 形成初始激活结构

知识单元仍然是局部知识。

系统可以从知识单元中形成轻量的初始激活结构。

它不是完整知识体系，也不是最终 知识结构。

它主要包括：

```text
知识单元中的可激活知识点；
可能匹配的已有领域；
可能匹配的已有概念；
候选领域和候选概念；
来源、证据和使用线索。
```

领域和概念不应该仅因为导入了一篇文档就稳定生成。

它们更多是在真实问题处理中被反复激活、使用、验证和反馈后逐渐稳定下来。

领域和概念是通用定义，其边界调整来自长期观察；认知视角的差异主要体现在使用过程中形成的 ActivationLink 上。同一批材料，在不同使用者、任务类型和问题场景下，可以形成不同的激活路径和实践沉淀，而不是各自长出完全不同的领域和概念体系。

因此，知识体系不是从导入文件时一次性建立，而是在使用中逐渐长出来。

ActivationLink 也不应在导入阶段稳定生成。它是在真实问题处理中，根据问题主题、认知视角和反馈逐渐形成的认知连接，让相关知识点可以被快速激活，并通过知识点反查知识单元和来源。Wiki 页面同样来自长期使用中的稳定主题，而不是导入时的直接产物。

## 3. 问题如何被处理

当用户提出问题时，系统不应该直接检索文档并生成回答。

它也不应该无论什么问题都进入完整推理链路。

它应该先识别问题需要哪种思维模式，再按对应方式处理。

### 理解问题

系统首先判断问题是什么。

它需要识别：

```text
问题领域；
问题类型；
核心概念；
用户意图；
需要的回答形态；
是否需要外部证据。
```

问题理解决定后续激活哪些长期记忆。

### 选择思维模式

理解问题之后，系统需要选择处理模式。

常见模式包括：

```text
直接记忆；
快速检索；
经验路径；
工作模型；
查证；
冲突检测；
反思学习。
```

模式选择决定后续是否需要激活长期记忆、查找外部证据、组织临时认知模型，或在产生学习价值时形成 Learning Event 并触发 Study。

知识大脑不是所有问题都完整推理。简单、熟悉、低风险的问题，可以直接基于召回并经 Rerank 分类后的相关证据组织回答；明确查找类问题，可以走快速检索；复杂设计、判断和决策问题，应该进入工作模型模式，按思维模板组织临时认知模型和推理；时效性强、风险高或存在证据冲突的问题，应该进入查证或冲突检测模式；回答失败或用户纠正时，应该进入反思学习模式。

### 分层检索与激活

激活长期记忆不是单一路径，而是分层展开。

系统优先沿使用中形成的 ActivationLink 激活知识点，并通过知识点反查知识单元和来源。当正式激活路径不足时，再依次借助材料目录结构树、全文检索或相近表达检索进行补充；当前知识不足、可能过期或风险较高时，再查找外部证据。

各层召回的结果不应按来源层级简单排序，而应统一比较相关性、证据质量、问题目标、历史使用效果和可追溯性，再决定采用哪些知识进入回答。

### 激活长期记忆

系统根据问题和已选择的模式激活相关知识。

激活不是全文搜索，而是沿着领域、概念、关系、经验和实践路径进行联想。

系统应该优先调用已经形成的长期知识结构。

### 判断知识缺口

系统需要判断已有知识是否足够。

它应该能区分：

```text
可以直接回答的部分；
需要证据确认的部分；
当前知识中缺失的部分；
存在冲突或不确定的部分。
```

这是系统的元认知能力。

### 查找外部证据

当长期记忆不足时，系统应查找外部证据。

外部查找不是随机搜索，而是由当前问题和知识缺口驱动。

它的目标是验证假设、补充事实、确认边界或发现反证。

### 组织临时认知模型

只有复杂问题才需要构建临时认知模型。

当问题足够复杂、存在多因素判断、需要冲突检测，或者召回并经 Rerank 分类后的相关证据不足以形成可靠回答时，系统需要把激活的长期记忆和查找到的外部证据组织成本次问题的临时认知模型，并按相应思维模板组织推理。

这个模型应该包含：

```text
当前问题；
核心变量；
相关知识；
已知事实；
缺失信息；
支持证据；
反对证据；
可能推理路径；
回答边界。
```

临时认知模型是回答生成前的关键中间层。

### 推理与回答

系统基于临时认知模型完成推理。

推理过程中需要检查：

```text
结论是否有证据支撑；
关系是否成立；
是否存在冲突；
是否需要表达不确定性；
回答是否解决用户问题。
```

最终回答是当前认知条件下的最优输出，而不是绝对真理。

## 4. 学习事件与长期记忆调整

Knowledge Brain 不追求保存每次问题处理的完整思考过程。

系统只在出现学习价值时记录 Learning Event。没有学习价值的问题处理，可以不生成 Trace，也不阻塞回答流程。

**有效学习不依赖用户持续纠正。** 主学习燃料来自每次问答中自然产生的检索事件：`activation_success`（路径命中且知识被采用）、`activation_failure`（路径命中但未支撑有效回答）、`activation_gap`（无合适路径但补充查找找到被采用的知识）。这些事件自动产生；Study 通过跨次累积（`repeated_success` / `repeated_failure`）驱动 ActivationLink 的形成、强化、降权和淘汰。`user_correction` 是高价值补充信号，可加速边界修正，但不是演化的前提。详见 `study.md` 第 2 节与 `trace.md`。

Learning Event 关注的事实信号包括：

```text
【检索事件——主驱动】
激活路径是否成功或被采用（activation_success）；
激活路径是否命中但失败（activation_failure）；
是否存在激活路径缺口且补充结果被采用（activation_gap）；
同类条件下是否反复成功或失败（repeated_success / repeated_failure）；

【其他学习信号】
是否存在认知缺口或知识冲突；
是否有候选 ActivationLink 值得观察；
是否需要调整 Concept 或 Wiki；
是否需要重新验证某个 KnowledgePoint；

【用户反馈——补充加速】
是否出现用户纠正（user_correction）。
```

Study 根据 Learning Event 调整长期记忆，尤其是 ActivationLink 的强化、降权、淘汰和候选形成；在结构信号足够稳定时，也可提炼候选实践路径供经验路径模式调用。详见 `study.md`。

完整复盘——例如为什么一次任务失败、如何改进行动策略——属于 Agent Runtime，不属于 Knowledge Brain 的核心职责。

## 5. 知识如何学习和演化

知识大脑必须从使用中学习。

每一次问题处理都会暴露长期记忆的状态——**主要通过检索与采用事实**，而不依赖用户是否纠正：

```text
哪些 ActivationLink 被召回、是否被实际采用；
哪些路径支撑了有效回答，哪些命中后仍失败；
哪些问题只能通过补充查找回答，暴露了激活缺口；
哪些知识真正参与推理；
哪些问题暴露材料或认知缺口。
```

系统应根据这些检索事件及其累积调整长期记忆。用户反馈可以加速修正，但不是学习的必要条件。

### 强化

被反复验证有效的知识点、ActivationLink 和实践路径，应该被强化。

实践路径的强化通常来自另一条链路：工作模型模式下某类问题的组织方式在多次 Learning Event 中被证明稳定有效（例如 activation_success 反复触发、核心变量组合高度相似），Study 将其提炼为候选实践路径，经独立验证后供经验路径模式调用。详见 `study.md` 第 7 节。

### 修正

导致错误理解或错误回答的知识、ActivationLink 或推理路径，应该被修正。

### 补充

反复暴露的缺口，应该成为新知识输入或新学习任务。

### 重组

当旧的知识结构无法解释新的问题模式时，系统应该允许知识结构重组。

### Wiki 编译

当某个主题在材料层和认知层都已足够稳定，表达层应将其编译为 Wiki 页面，或在已有页面上补充、重组和更新。Wiki 编译是学习的结果，不是导入的副产品。

知识演化不是异常流程，而是知识大脑持续成长的正常方式。

## 6. 生命周期与记忆有效性

长期记忆不是永远有效。

知识可能会过期、被修正、被替代或失去上下文。

因此，系统需要管理记忆有效性。

当前回答应该优先使用有效、稳定、可追溯的知识。

旧知识可以保留，用于解释历史思考和知识演化，但不应该默认作为当前答案依据。

生命周期管理在知识大脑中相当于记忆的有效性判断和重新验证机制。知识状态应细分为 current、candidate、needs_verification、conflicted、superseded、deprecated、historical 和 retracted，而不是只用有效、过期、历史三类粗粒度标签。

## 7. 认知稳定性与质量控制

知识大脑不仅要能吸收和使用知识，还要防止错误路径、临时思考和过期结论被误当作长期真理。

系统在多个层面需要质量控制：

**ActivationLink 需要验证、降权和淘汰**

链接不是「被使用过」就会自动稳定。candidate 只能辅助探索，verified 需要持续校验，weakened、conflicted 和 deprecated 都不应作为当前首选激活路径。Study 的目标是筛选和淘汰链接，而不是让链接数量无限增长。

**思维模式选择需要由认知维度约束**

模式选择应基于确定性、风险、时效性、证据充分性、冲突度、复杂度和用户意图，而不是完全依赖模型主观判断。简单问题可轻量处理，复杂问题才进入 Working Model，高风险或冲突问题必须进入查证或冲突检测。

**Working Model 需要质量检查**

临时认知模型在输出前应检查问题边界、变量完整性、证据覆盖、缺口显式性、正反证据并存、推理可回溯性，以及回答边界是否与证据强度匹配。Working Model 是一次性思考结构，不能直接沉淀为长期记忆。

**知识生命周期需要支持冲突、候选、废弃、撤回**

长期记忆应区分 current、candidate、needs_verification、conflicted、superseded、deprecated、historical 和 retracted。conflicted 不等于错误；historical 和 retracted 不应默认参与当前回答；current 也不是永久状态。

**Wiki 编译需要防止长期结论固化**

Wiki 页面是当前阶段的稳定表达，不是最终真理。页面应保留适用边界、争议点、待验证点、证据来源和修订记录；新证据和底层状态变化应触发重新编译，冲突知识不能被强行合并成单一结论。

## 认知预算

知识大脑不应对所有问题都执行完整认知流程。

认知预算用于控制：

```text
检索深度；
KPN 扩展深度；
Working Model 规模；
查证强度；
冲突检测强度；
Study 深度。
```

认知预算不是工程性能参数，而是认知设计约束。它约束系统「愿意花多少认知资源」处理当前问题，避免过度思考和无限扩展。

### 三类预算

**low**：简单、低风险、证据直接充分的问题。

**medium**：需要多证据组织、轻量判断、有限上下文补充的问题。

**high**：复杂设计、重要决策、高风险、高冲突、高时效性问题。

### 预算与处理深度

```text
low 通常不进入 Working Model；
medium 可以进入轻量 Working Model；
high 才进入完整 Working Model、查证和冲突检测。
```

预算不足时，应优先保证证据可靠和回答边界，而不是强行扩展知识、继续扩散 KPN 或堆叠推理步骤。

当 low budget 下证据已直接充分，应停留在证据直答；当 high budget 下仍无法形成可靠结论，应输出边界和缺口，而不是无限扩展检索或 KPN。

认知预算与思维模式选择共同决定问题处理深度，详见 `cognitive-routing.md`。

## 8. Agent 如何使用知识大脑

Agent 平台和知识大脑的职责不同。

Agent 平台负责执行任务。

知识大脑负责提供知识、证据、思维模式、临时认知模型、学习事件记录和学习沉淀。

因此，Agent 不应该直接访问整个知识仓库，也不应该把知识大脑当成一个普通 RAG 工具。

它应该在明确领域边界内调用知识大脑能力。

知识大脑可以为 Agent 提供：

```text
领域长期记忆；
相关知识激活；
来源证据；
思维模式选择；
临时认知模型；
冲突检测；
Learning Event 记录；
经验沉淀和学习建议。
```

Agent 使用知识时，也应该经历类似过程：

```text
理解任务；
选择思维模式；
激活领域记忆；
判断缺口；
必要时组织临时认知模型；
推理执行；
若产生学习价值则形成 Learning Event；
反馈学习。
```

Agent 不是知识系统之外的消费者，而是知识大脑的一种实践入口。

Agent 的使用结果也应该反哺长期记忆。

但知识大脑不负责 Agent 平台本身的通用能力，例如：

```text
任务调度；
工具运行时；
浏览器或终端操作；
代码编辑执行；
多 Agent 协作协议；
权限和沙箱管理。
```

这些能力属于 Agent 平台。

知识大脑只关注知识如何被这些任务使用，以及使用之后如何被追踪、评估和沉淀。

## 9. 能力对应关系

系统可以按认知能力组织。

```text
外部材料进入系统；
材料被整理成可追溯的知识单元和知识点；
使用中形成候选 ActivationLink；
问题触发合适的思维模式（简单问题轻量处理，复杂问题进入工作模型）；
优先沿 ActivationLink 激活记忆，不足时分层补充查找；
必要时组织临时认知模型；
若产生学习价值则形成 Learning Event，Study 调整长期记忆；
长期稳定的主题编译为 Wiki 页面；
长期记忆的状态和有效性被持续管理。
```

这些能力不是孤立工具，而是知识大脑认知链路中的不同部分。

实现时应避免把认知能力直接做成由模型自由调用的 skill。

认知能力涉及知识库、索引、状态、证据、错误处理和学习事件记录，应该由稳定 service 承载。

Agent 根据思维模式编排这些 service。

模型只在需要语义判断、摘要、匹配或推理时，作为 service 内部步骤被调用。

## 10. 总结

系统设计的核心不是“文档如何检索”，而是“知识如何被一个持续学习的认知系统使用”。

整体链路是：

```text
材料进入
  -> 形成知识单元和知识点
  -> 使用中形成或验证 ActivationLink
  -> 问题触发思维模式选择
  -> ActivationLink 激活知识，不足时分层补充查找
  -> KPN 在必要时补充上下文
  -> Working Model 在复杂问题中组织思考
  -> 生成回答

如果本次问答产生学习价值：
  -> 形成 Learning Event
  -> Study 根据 Learning Event 调整长期记忆
  -> 稳定结果可进入 Wiki 编译
  -> 长期记忆和认知结构持续演化
```

这就是知识大脑系统的基本设计。
