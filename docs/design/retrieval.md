# 知识激活与证据检索

在知识大脑中，检索不是简单搜索片段。

它包含两件事：

```text
激活长期记忆；
查找外部证据。
```

长期记忆激活对应人的内部联想。

外部证据检索对应人在不确定时查资料、问工具、看原文。

这两件事不是冷启动和热启动两套流程。

系统始终使用当前认知视角下已有的领域和概念理解问题并激活知识，同时用多种方式查找材料和证据作为补充。

如果领域和概念没有激活出有效答案，但补充查找找到了相关知识并最终被采用，这说明当前认知结构存在缺口。

这个缺口可形成 Learning Event，供 Study 识别候选领域和候选概念，而不是立即变成稳定知识体系。

检索不是所有问题的第一步。

它由思维模式选择触发：直接记忆模式可能不需要检索，快速检索模式只需要轻量查找，工作模型和查证模式才需要更完整的知识激活与证据检索。

对于需要检索的问题，系统先采用统一的召回基础，再根据证据是否能够直接回答问题选择回答深度：

```text
问题拆解
  -> 目录结构召回 + FTS 等多路召回
  -> Rerank（缩小证据集合）
  -> 直接回答充分性判断
       -> 充分：证据直答
       -> 不充分：结构化证据处理 -> Working Model -> 受约束推理
```

证据直答是默认在线路径。系统把问题和 Rerank 筛选后的证据交给回答模型，由模型从现成证据中组织答案；模型只返回所使用的稳定 `fact_id`，代码负责校验白名单、回填 `EvidenceRef` 并保证答案可追溯。定义、角色或事项枚举、属性、材料定位、明确规则和简单流程等问题，只要证据中已经包含答案，都应走这条路径。

结构化证据聚合不是所有问题的必经步骤。只有当召回证据不能直接形成答案，确实需要多跳关系、跨材料组合、冲突消解、隐含条件推导、复杂比较或高风险判断时，才升级到结构化深路径。问题表面出现“判断”“比较”“为什么”等词，不足以单独触发升级；回答是否需要原因、比较或推理步骤，也不由 Rerank 决定。

### Rerank 的职责边界

多路召回以召回率为目标，允许重复、弱相关和背景材料进入候选集。

Rerank 只负责缩小提供给回答模型的证据集合。它根据当前问题，从候选事实中选出最适合交给回答模型的证据，并尽量保持相关事实在候选集中的相对顺序。

Rerank 不负责理解完整回答结构，也不负责决定回答是否需要原因、比较或推理步骤。这些由回答模型、结构化证据聚合或 Working Model 承担。

对每条候选证据，Rerank 只判断两件事：

```text
是否与当前问题匹配；
证据角色：直接证据、支持性证据，或无关证据。
```

只有直接证据和必要的支持性证据进入后续环节；无关证据应被排除。Rerank 不抽取或陈述证据支持的事实、实体或关系，也不按问题类型预定义槽位或答案形态——这些属于结构化证据聚合、覆盖检查或 Working Model 的工作。

在深路径中，结构化证据聚合负责把证据转换为问题所需的事实形态。例如：

```text
定义型：概念 -> 定义 -> 限定条件；
集合型：实体 -> 职责或属性 -> 证据；
流程型：步骤 -> 条件 -> 执行主体 -> 证据；
比较型：比较维度 -> 对象事实 -> 差异；
关系型：实体 A -> 关系 -> 实体 B -> 证据；
原因型：结论 -> 原因链 -> 支撑证据。
```

短路径只做直接回答充分性和引用有效性检查，不预先把角色、职责等完整陈述拆成实体关系再聚合。深路径才执行槽位覆盖和冲突检查。两条路径的最终答案都只能使用已召回证据，并且引用必须绑定到实际支撑答案事实的证据。

是否进入推理不能只由问题类型决定，而应以召回后的证据状态决定。例如“如何判断是否继续投入”如果证据已明确列出判断规则，应直接组织答案；只有规则需要组合或推导时才进入推理。在线回答产生的学习信号和认知结构优化应异步处理，不应阻塞证据直答。

## 1. 检索由思维模式决定

系统应先判断当前问题需要哪种处理方式。

不同模式下，检索的深度不同：

```text
直接记忆模式：尽量不检索，只调用稳定长期记忆；
快速检索模式：查找明确来源或片段；
经验路径模式：优先激活已经沉淀的实践路径；
工作模型模式：激活相关知识并查找支撑证据；
查证模式：优先寻找当前、可靠、可引用的外部证据；
冲突检测模式：主动寻找支持和反对证据。
```

因此，检索不是固定流程，而是被思维模式调度的能力。

## 2. 检索从问题理解和推理模板开始

系统不能在没有理解问题的情况下直接搜索。

它需要先判断：

```text
问题属于哪个领域；
涉及哪些概念；
是在问事实、方法、判断还是设计；
已有长期记忆是否可能足够；
是否需要外部证据。
```

问题理解决定检索方向。

对于需要推理的问题，问题理解和知识检索之间必须存在正式的推理模板层。模板不是回答阶段才拼接的 Prompt，而是检索阶段的输入：

```text
ProblemFrame -> ReasoningPatternPlan -> EvidenceSlot[]
             -> slot-aware recall -> Rerank -> PatternCompleteness
```

召回可以按证据槽位定向展开；Rerank 仍只做问题级证据筛选，不按模板槽位理解完整回答结构。`PatternCompleteness` 检查 Rerank 后的证据是否满足模板所需的结构完整性，这一检查不属于 Rerank。

每个模板至少定义适用问题类型、证据槽位及必需性、证据质量要求、带前置条件的推理步骤、输出契约，以及缺失、冲突和外部查证时的行为。

检索结果不能只提供知识单元总相关分，还应提供槽位覆盖：

```json
{
  "unit_id": "unit_x",
  "slot_matches": [{
    "slot_id": "governing_rules",
    "score": 0.91,
    "fact_ids": ["fact_1"]
  }]
}
```

槽位覆盖必须检查来源权威性、实体绑定、时间有效性和证据状态。系统必须区分 `not_retrieved`、`not_available` 与 `requires_external_verification`，不能把检索失败和材料不存在混为一谈。

这里的问题理解基于当前认知视角。

系统会先按自己已有的领域和概念去理解问题，就像一个刚进入真实工作环境的学生会先用已经学过的知识体系解释问题。

这个体系可能不足，但系统不切换到另一套“冷启动流程”。

它仍然按当前知识体系工作，同时依靠补充查找暴露和补救认知缺口。

## 3. 问题需要先拆成认知路由上下文

问题理解不能只得到几个关键词。

系统需要把用户问题和待处理事项拆成一组可用于路由的认知维度：

```text
场景；
目标；
思维模式；
注意焦点；
知识点相关性；
历史反馈；
认知预算。
```

这些维度共同构成本次检索的认知路由上下文。

可以理解为：

```text
原始问题
  -> 场景识别
  -> 目标识别
  -> 模式选择
  -> 注意焦点提取
  -> 相关知识线索
  -> 历史反馈约束
```

场景说明当前问题发生在什么环境中。

例如：

```text
投资；
故障排查；
系统设计；
文档查找；
学习总结；
决策评估。
```

目标说明用户希望达成什么结果。

例如：

```text
解释；
判断；
选择；
定位原因；
生成方案；
验证事实；
总结经验。
```

思维模式说明系统应该使用哪类处理程序。

例如：

```text
快速检索；
工作模型；
查证；
冲突检测；
经验路径；
反思学习。
```

注意焦点说明本次问题应该优先关注哪些维度。

例如：

```text
风险；
成本；
增长；
性能；
根因；
证据；
边界；
执行步骤。
```

知识点相关性说明哪些知识内容和本次问题直接相关，哪些只是背景。

历史反馈说明过去相似问题中哪些激活路径有效、无效、被采用或被否定。

因此，检索入口不应该只是：

```text
query text
```

而应该是：

```text
CognitiveRouteContext
  scene
  goal
  pattern
  focus
  knowledge_relevance
  feedback_context
  cognitive_budget
```

这个上下文会同时影响领域匹配、概念匹配和激活链接匹配。

## 4. 先匹配领域和概念，再匹配激活链接

领域和概念是认知入口，但它们仍然是相对粗粒度的入口。

如果一个概念下面挂载了大量激活链接，命中概念后直接遍历所有链接，会导致两个问题：

```text
效率低；
质量下降。
```

因为同一个概念下的知识不一定都适合当前场景、目标和思维模式。

因此，认知结构检索应该分成两层匹配：

```text
领域 / 概念匹配；
激活链接匹配。
```

更完整的链路是：

```text
问题
  -> 认知路由上下文
  -> 匹配 Domain
  -> 匹配 Concept
  -> 匹配 ActivationLink
  -> 激活 KnowledgePoint / KnowledgeUnit
```

`ActivationLink` 不应该只表达“这个知识点属于这个概念”。

它更应该表达：

```text
在某个认知视角下，
当问题处于某类场景、目标、思维模式和注意焦点时，
可以通过这个概念激活这个知识点。
```

因此，`ActivationLinkMatcher` 需要判断：

```text
这条链接是否适合当前场景；
这条链接是否服务当前目标；
这条链接是否匹配当前思维模式；
这条链接是否覆盖当前注意焦点；
链接指向的知识点是否和问题相关；
过去类似问题中这条链接是否有效。
```

最终不应该展开概念下的全部链接，而应该选择最相关的一组链接。

ActivationLink 不是「被使用过」就会自动稳定。正式召回时，系统应依据链接状态和有效性判断，而不是只看历史命中次数。

### 正式召回规则

正式召回应遵循以下原则：

```text
优先使用 verified 链接；
candidate 链接只能用于补充探索，不能直接决定答案；
conflicted 链接必须触发冲突检测；
weakened 和 deprecated 链接不应作为当前首选激活路径。
```

verified 链接可参与正式召回，但仍需在本次问题中重新检验场景匹配度、答案贡献度和边界适用性。verified 不等于永久有效。

candidate 链接不得进入正式召回候选集，不得作为低权重激活路径替代 verified 链接。它只能提示「可能存在值得验证的路径」，并交由补充查找、Learning Event 和 Study 独立验证。

conflicted 链接一旦命中，不应直接采用其指向知识，而应进入冲突检测：比较支持证据、反对证据、边界条件和替代路径，再决定采用、保留不确定性或降权废弃。

这里的「链接」在正式召回中主要指 verified 状态的 ActivationLink。

例如：

```text
Concept matched
  -> ActivationLinkMatcher
  -> top links
  -> KnowledgePoint / KnowledgeUnit
```

这种方式让认知结构检索更接近人的思考。

人不是看到“投资”这个概念后遍历全部投资知识，而是先判断当前是在做成长判断、估值判断、风险控制还是趋势交易，再激活对应知识。

## 5. 内部知识激活

内部知识激活不是全文搜索。

它是沿着长期记忆中的领域、概念、知识点、关系、经验和实践路径进行联想。

例如，用户问“知识检索方式是什么”，系统应该激活：

```text
知识大脑；
长期记忆；
知识索引；
临时认知模型；
证据回溯；
学习事件记录；
反馈学习。
```

这些被激活的知识为本次问题提供初始理解。

内部激活应优先于补充查找发生。系统先尝试人已经形成的「熟路」，也就是当前认知视角下 verified 的 ActivationLink；只有 verified 路径没有返回足够、相关且可验证的知识时，才扩大到补充查找。

补充查找包括：

```text
从知识单元中查找；
从知识点中查找；
从原文和目录线索中查找；
从来源位置回到证据；
从相近表达中寻找可能相关的材料。
```

领域、概念和正式激活关系是当前知识体系的主线。

补充查找是发现证据和认知缺口的机制。

候选激活关系不承担召回职责。补充查找独立找到并采用某个知识单元后，学习流程才可以用该结果验证已有候选关系：验证通过且满足有效性标准时晋升为 verified；验证不通过、边界不清或存在更优替代路径时，转为 weakened 或 deprecated。

## 分层检索路径

知识大脑的检索不是单一路径，而是分层进行。

系统按认知路径的成熟度和材料侧的结构线索，依次展开不同层级的查找：

```text
第一层：ActivationLink 激活链接；
第二层：目录结构树；
第三层：全文检索或相近表达检索；
第四层：必要时外部证据检索。
```

这三层内部查找加上外部证据，承担不同职责，不应混为一谈。

ActivationLink 是使用中形成的认知路径。它记录过去反复激活、实际采用和反馈强化后沉淀下来的主题关联，代表系统已经学会如何面对某类问题。

目录结构树是材料原有表达结构的保留。它来自规范化 Markdown 中的标题、章节和层级关系，反映作者或材料本身的组织方式，而不是系统在使用中形成的认知结构。

全文检索或相近表达检索是兜底补充查找。当正式激活路径和目录线索都不足以覆盖当前问题时，系统才沿关键词、全文索引或语义相近表达在材料中广泛搜索，防止遗漏。

外部证据检索用于当前知识不足、可能过期或风险较高的情况。它带着明确假设查找材料内外的补充依据，用于验证、补事实或发现反证。

优先使用 ActivationLink 的原因在于：

```text
它携带过去使用中形成的主题关联；
它可以通过知识点快速反查知识单元；
它更接近人的长期记忆激活。
```

人面对熟悉问题时，往往先沿已经形成的联想路径回忆，而不是从整本书的第一个字开始翻找。ActivationLink 对应的就是这种已经验证过的激活路径。

目录结构树的价值在于：

```text
保留作者或材料原有组织；
在 ActivationLink 不成熟时提供结构化补充；
有助于避免只依赖关键词或向量相似度。
```

当某份材料尚未被充分使用、相关 ActivationLink 还未稳定时，目录结构仍能提供有意义的范围限定。它让补充查找沿着材料的自然章节展开，而不是在全文平面上盲目匹配。

全文检索的价值在于：

```text
防止遗漏；
发现尚未形成 ActivationLink 的新知识；
暴露当前认知结构缺口。
```

如果 ActivationLink 已经稳定且命中充分，全文检索可能只需要轻量参与。但如果正式路径未命中、目录线索也不足，全文检索就是发现新材料和认知缺口的重要机制。被反复采用但尚未沉淀为 ActivationLink 的结果，可形成 Learning Event，供 Study 后续处理。

分层检索并不意味着结果按来源层级简单排序。

来自 ActivationLink 的结果不一定天然优于目录召回或全文检索结果。系统应对各层召回的内容统一比较：

```text
与当前问题的相关性；
证据质量和可追溯性；
是否服务当前问题目标；
历史使用效果；
是否适合进入本次回答。
```

一层路径找到候选结果，只说明它值得被纳入比较，不说明它必然优先采用。最终采用哪些知识，仍取决于它们能否真正支撑本次回答。

## KnowledgePoint Network 在检索中的作用

KPN 不作为第一检索入口。

系统不应该从用户问题直接进入 KPN 遍历，也不应该把 KPN 当成另一套图搜索。

正确流程是：

```text
问题
  -> 思维模式选择
  -> ActivationLink / 目录结构 / 全文检索找到核心 KnowledgePoint
  -> KPN 围绕核心 KnowledgePoint 做局部扩散
  -> 补充必要上下文
  -> 进入证据直答或 Working Model
```

KPN 的使用边界是：

```text
它只能围绕已经召回或采用的核心 KnowledgePoint 做局部扩散；
它不负责全局搜索；
它不参与 candidate ActivationLink 的正式召回；
它不根据图距离直接生成答案；
它只补充上下文、变量、边界、反例和相关知识。
```

KPN 在检索中主要承担四类作用：

### 1. 补充上下文

当 ActivationLink 找到核心知识点后，KPN 可以补充与该知识点紧密相关的上下文知识。

例如：

```text
OAuth2
  -> Access Token
  -> Authorization

OIDC
  -> ID Token
  -> Authentication
```

这些上下文不一定是问题字面命中的内容，但可能是理解核心知识点所必需的背景。

### 2. 辅助 Rerank

当多个候选 KnowledgePoint 文本相似度接近、Rerank 难以仅凭字面判断匹配度时，KPN 可以提供结构相关性信号，帮助判断某条候选是否匹配当前问题、应归为直接证据还是支持性证据。

可参考：

```text
是否与核心 KnowledgePoint 直接相连；
连接类型是否与当前问题相关；
连接强度是否来自历史采用；
是否属于理解核心知识点所必需的上下文，而不是泛相关背景；
是否存在反向或对立关系。
```

KPN 只提供辅助信号，不能单独决定最终采用，也不能替代 Rerank 对证据角色的判断。

### 3. 暴露知识缺口

如果核心 KnowledgePoint 的邻近知识点缺失，或者某个必要上下文在系统中不存在，应记录为知识缺口。

例如：

```text
当前问题需要 OAuth2、OIDC、JWT；
系统只找到 OAuth2 和 OIDC；
KPN 判断 JWT 是必要上下文但不存在；
则形成 Knowledge Gap。
```

该缺口可形成 Learning Event，并由 Study 判断是否需要补充材料或形成新的学习目标。

### 4. 支持 Working Model 构建

复杂问题进入 Working Model 时，KPN 可以帮助识别核心变量、边界条件、前置知识和反例。

例如投资问题中，核心知识点是「创业板」，KPN 可以扩散出：

```text
估值；
利润增速；
ROE；
流动性；
风险偏好；
政策环境。
```

这些内容不是直接答案，而是 Working Model 的变量候选。

KPN 的激活边界、扩展边界和停止条件详见 `docs/design/kpn.md`。

### KPN 检索边界

KPN 在检索中的边界可概括为：

```text
ActivationLink 是主激活路径；
KPN 不是召回入口；
KPN 只在核心 KnowledgePoint 已激活后，为 Working Model 补充上下文。
```

KPN 扩展原则：

```text
不因相关就扩展；
只围绕变量、边界、反例、缺口扩展；
不参与简单证据直答；
不替代目录结构召回、全文检索、外部证据查找。
```

### 认知预算与检索深度

检索深度由问题需求、证据状态和认知预算共同决定。

不是所有问题都需要扩展 KPN，也不是所有问题都需要 Working Model。

```text
low budget：
  目录结构召回 + FTS + Rerank，直接证据充分即可回答；
  通常不激活 KPN。

medium budget：
  可以补充有限 ActivationLink 召回和必要 KPN 上下文；
  可进入轻量 Working Model。

high budget：
  可以进入 Working Model、冲突检测、外部查证和更完整证据覆盖；
  KPN 扩展仍受 Working Model 目标和停止条件约束。
```

预算不足时，检索应优先保证证据可靠和可追溯，而不是继续扩大召回范围或无限扩散 KPN。

## 6. 认知结构检索和补充查找不是竞争关系

认知结构检索和补充查找不应该被理解为两套互相竞争的检索方式。

它们的关系更接近：

```text
补充查找负责不要漏；
认知结构负责更会用。
```

补充查找可以从材料、知识单元、知识点、目录线索、来源位置和相近表达中寻找可能相关的内容。

认知结构则提供问题理解、结果筛选、答案组织和学习判断的框架。

因此，两者都参与问题处理，但有明确的先后次序和职责边界。

系统不应该简单地认为“来自领域和概念的结果一定优先”，也不应该让补充查找完全覆盖当前认知结构。

更合理的方式是：

```text
先用当前认知结构理解问题；
优先匹配 verified ActivationLink，并直接取回其 KnowledgePoint / KnowledgeUnit；
验证正式路径返回内容的回答度、适用边界和证据质量；
命中 conflicted 链接时，进入冲突检测后再决定是否采用；
正式路径不足时，再启动知识单元、知识点、目录和原文等补充查找；
candidate 链接只可作为补充探索线索，不能直接决定答案；
用补充结果完善回答、发现认知缺口，并独立验证 candidate ActivationLink；
最终只采用能支撑本次回答的知识。
```

也就是说，认知结构检索的优越性不只是“找到更多结果”。

它的价值在于提升知识使用质量。

更准确地说，认知结构检索是一层知识利用质量提升机制。

它让系统在一次次使用中不断判断：

```text
问题是否被放进了正确领域；
核心概念是否被激活；
召回结果是否被正确区分主次；
答案是否围绕稳定判断展开；
补充查找是否暴露了认知结构缺口；
相似问题下是否逐渐形成更稳定的使用路径。
```

因此，认知结构越成熟，系统不只是更容易找到材料，也会更稳定地使用材料。

## 7. 认知结构如何提升检索质量

认知结构至少在三个阶段发挥作用。

第一，在检索前帮助理解问题。

它让系统判断：

```text
这个问题属于哪个领域；
涉及哪些概念；
哪些概念容易混淆；
需要哪些前置判断；
答案应该从哪些维度展开。
```

没有认知结构，系统只能从字面相似性出发。

有了认知结构，系统可以带着领域、概念、边界和问题意图去寻找材料。

第二，在检索中帮助筛选和排序。

补充查找可能找到很多看起来相关的内容。

认知结构可以帮助判断：

```text
哪些结果命中了核心概念；
哪些结果只是背景材料；
哪些结果是边界条件；
哪些结果虽然文字相似但问题方向不一致；
哪些知识在过去处理相似问题时被证明有用。
```

因此，最终排序不应该只看文本相似度。

它还应该考虑当前认知视角、领域匹配、概念匹配、历史使用、来源质量和本次问题目标。

更进一步，排序应该考虑本次认知路由上下文：

```text
场景是否一致；
目标是否一致；
思维模式是否一致；
注意焦点是否一致；
知识点是否直接支撑本次问题；
历史反馈是否证明该路径有效。
```

第三，在检索后帮助组织答案。

补充查找可以提供材料，但不会天然形成好的回答结构。

认知结构可以帮助系统决定：

```text
答案应该围绕哪个核心概念展开；
哪些知识是主证据；
哪些知识只是补充；
哪些内容用来说明边界；
哪些内容提示风险或例外；
是否需要指出当前认知结构存在缺口。
```

所以，认知结构不是补充查找的替代品。

它让补充查找找到的材料被更好地理解、筛选和使用。

## 8. 缺口判断

激活长期记忆后，系统需要判断是否足够。

它应该区分：

```text
可以直接使用的知识；
需要进一步确认的知识；
缺失的事实；
可能存在冲突的地方；
需要外部资料支撑的判断。
```

这一步体现系统的元认知。

特别地，如果出现以下情况：

```text
当前领域和概念激活失败或找到的知识质量低；
补充查找找到了高相关知识单元或知识点；
这些结果进入临时认知模型或最终回答。
```

那么系统应记录认知缺口。

如果当前领域和概念也找到了结果，但补充查找找到了更好的结果，也不应该把这看成两者竞争失败。

这说明当前认知结构可能还不够精确：

```text
已有概念没有覆盖这个问题；
已有激活路径没有连接到关键知识；
已有领域边界太宽或太窄；
过去使用沉淀出的路径不适合当前问题。
```

认知缺口可能意味着：

```text
缺少某个领域；
缺少某个概念；
缺少某个思维模式；
已有概念边界不对；
已有领域不能解释当前问题；
某条激活路径需要调整。
```

这些缺口不会立即修改长期记忆，而是可形成 Learning Event，交给 Study 处理。

## 9. 外部证据检索

当长期记忆不足时，系统应查找外部证据。

外部检索不是盲目搜索，而是带着问题和假设查找。

系统应该知道自己为什么要查：

```text
为了验证一个判断；
为了补充一个事实；
为了找到反对证据；
为了确认当前知识是否过期；
为了补足临时认知模型中的缺口。
```

## 10. 证据回到来源

被采用的知识应该能回到来源。

系统需要区分：

```text
来自长期记忆的概念；
来自知识单元的证据；
来自外部材料的新信息；
系统根据关系组织出的推理。
```

这样回答才不会变成无来源的断言。

## 11. 检索服务于临时认知模型

检索的结果不是直接交给回答生成。

在工作模型、查证和冲突检测等模式下，它应该进入临时认知模型。

在快速检索模式下，检索结果也可以直接形成简短回答或来源列表。

临时认知模型会决定：

```text
哪些知识真正相关；
哪些证据被采用；
哪些内容只是候选；
哪些地方存在冲突；
应该如何组织推理路径。
```

因此，检索的目标不是返回最多材料，而是为本次问题提供合适的认知材料。

## 12. 检索与 Learning Event

检索阶段默认不产生 Trace。

检索的首要目标是找到可回答问题的证据，而不是生成解释性 Trace。Trace 不应影响在线检索路径和性能。

只有当检索结果暴露长期记忆学习价值时，才形成 Learning Event。常见触发情况包括：

```text
ActivationLink 命中并被采用；
ActivationLink 命中但失败；
没有 ActivationLink，但通过目录结构、全文检索或外部证据找到有效知识；
检索结果暴露知识缺口；
检索结果暴露知识冲突；
用户反馈表明检索路径错误。
```

检索过程不记录：

```text
每个召回步骤；
每个候选结果；
rerank 详细解释；
每个 KPN 扩展节点；
Working Model 的内部组织过程。
```

Learning Event 只保留与学习相关的最小事实，供 Study 判断 ActivationLink、Concept 和知识状态是否需要调整。详见 `trace.md` 和 `study.md`。

## 13. 总结

检索系统的职责应分层：

```text
目录结构召回和 FTS 召回负责尽量不漏；
Rerank 负责按问题缩小证据集合，并标注直接证据、支持性证据或无关证据；
问题类型策略辅助短路径筛选证据，并为深路径提供结构化要求；
短路径检查直接回答充分性，深路径 Coverage 检查结构完整性；
受限补召回负责补齐缺失槽位；
结构化证据聚合和 Working Model 负责理解回答结构与推理组织；
Working Model 和 Reasoning 只消费已经正确组织的证据。
```

定义问题应选择可回链的直接定义；枚举问题应覆盖不同实体而非只取最高分；条件判断应以主规则、条件和结果为核心，例外不能替代主规则；关系问题必须同时具备两侧实体和显式关系或完整证据链。权重和相关分只用于辅助排序，不能替代这些结构性约束。

知识检索不是：

```text
搜索文本 -> 返回片段
```

而是：

```text
理解问题
  -> 选择思维模式
  -> 构建认知路由上下文
  -> 激活长期记忆
  -> 匹配领域、概念和激活链接
  -> 同时补充查找材料和证据
  -> 共同筛选和排序
  -> 判断知识缺口
  -> 支撑临时认知模型
  -> 若产生学习价值则形成 Learning Event
```

一句话总结：

```text
检索是知识大脑围绕问题激活记忆、寻找证据、支撑思考的过程。
```

检索路径的成熟度本身也是学习对象。当 ActivationLink、目录路径或补充查找方式暴露学习价值时，应形成 Learning Event，供 Study 判断哪些路径值得强化、降权或重组，而不是记录完整检索过程。
