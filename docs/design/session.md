

# **会话四层状态模型设计方案**

## **1. 设计目标**

本方案用于解决 AI 问答系统中的信息收集问题，包括：

- 用户表达不完整；
- 用户使用“它”“这个”“继续”等指代表达；
- 用户意图模糊；
- 用户补充信息仍然不清晰；
- 用户拒绝澄清或不继续澄清；
- 多轮对话中上下文不断变化；
- 问答系统需要在“追问”和“直接执行”之间做判断；
- 用户随时插话、切换话题，聊完别的之后希望系统记得刚才做到哪；
- 对话越长，早期信息越不应干扰当前判断；
- 同样程度的“不确定”，对不同操作的风险完全不同；
- 用户需要随时喊停、清空重来，而不被系统继续追问或长篇输出。

核心目标不是让模型每次都重新理解完整对话，而是建立一套显式状态机制，使系统能够稳定判断：

当前用户的问题是否可以回答？  
如果不能回答，到底缺什么？  
如果用户不澄清，系统应该如何安全推进？

在此基础上，四层模型还需具备四项“秘书级”能力，才能把单线流程升级为真正懂轻重缓急的会话状态机：

| 能力 | 解决什么问题 |
| --- | --- |
| 插话与召回 | 用户开小差聊别的，聊完后系统能主动回到搁置中的事 |
| 遗忘与淡化 | 长期对话里，陈旧信息自动降权乃至退出候选，聚焦当下 |
| 置信度与风险协同 | 低风险可默认推进，高风险哪怕很有把握也要再确认 |
| 系统级打断 | “别说了”“清空重来”等指令立刻掀桌子，回到干净待命态 |

---

# **2. 核心设计思想**

## **2.1 对话历史不是状态，只是证据**

传统做法容易把完整聊天记录直接交给模型，让模型自己判断上下文。

这种方式的问题是：

- 对话越长，理解越不稳定；
- 容易受到旧上下文干扰；
- 用户一句“继续”，模型很难稳定知道继续什么；
- 模型会反复追问已经问过的问题；
- 状态不可控、不可审计、不可复用。

因此，本方案采用：

对话历史只是状态更新的证据来源，真正驱动问答的是显式维护的状态。

也就是说，系统每轮对话后，都要把用户输入转化为结构化状态变化，而不是只依赖原始聊天记录。

---

## **2.2 先解析上下文，再判断是否缺信息**

用户表达不清晰时，不应该立刻追问。

例如：

把它删掉。

系统应该先尝试解析：

- “它”是否指向上一轮提到的文档；
- 是否指向当前正在处理的知识单元；
- 是否指向当前任务中的某个对象；
- 是否指向最近一次检索结果。

只有当上下文解析失败，或者置信度不足时，才进入澄清流程。

因此处理顺序应为：

```text
用户输入
  ↓
意图识别
  ↓
上下文解析
  ↓
信息完整性判断
  ↓
回答 / 执行 / 追问 / 安全默认推进
```

而不是：

```text
用户输入不清晰
  ↓
直接追问
```

---

## **2.3 追问不是默认动作，而是最后手段**

很多问答系统体验差，是因为遇到任何不确定性都追问用户。

更合理的策略是：

```text
能从上下文确定 → 直接处理
能给出候选 → 让用户选择
能安全默认 → 明确默认假设后继续
风险较高 → 必须追问
无法推进 → 说明无法判断
```

追问应该满足三个条件：

1. **当前信息确实不足**；
2. **系统无法从已有上下文补齐**；
3. **继续执行会产生明显错误或风险**。

---

# **3. 四层状态模型**

建议将系统状态分为四层：

```text
User Input
   ↓
Dialogue State
   ↓
Working State
   ↓
Knowledge State
   ↓
Answer State
```

四层分别解决不同问题。

---

# **4. 第一层：Dialogue State，对话状态**

## **4.1 定位**

Dialogue State 负责管理当前对话中已经确认的信息、待确认的信息、用户表达的意图和澄清历史。

它回答的问题是：

用户这轮话，在当前对话中意味着什么？

---

## **4.2 管理内容**

Dialogue State 主要包括：

```text
当前主题
当前意图
已确认信息
待确认信息
用户最近提到的对象
用户最近确认的对象
澄清问题历史
用户是否拒绝澄清
```

例如：

```text
当前主题：知识库检索
当前意图：查询设计缺陷
已确认信息：目标文档为 design.md
待确认信息：评估角度尚未明确
最近对象：design.md
澄清历史：已问过“从工程角度还是设计角度评估？”
```

---

## **4.3 设计原则**

Dialogue State 不保存所有聊天内容，只保存对后续理解有用的信息。

重点不是“记住用户说过什么”，而是：

提炼出当前对话继续推进所需的最小状态。

---

## **4.4 信息的鲜活度与淡化**

Dialogue State 不应把“曾经提过”等同于“现在仍 relevant”。

每进入一轮新对话，未被再次触及的信息都会逐渐失去影响力。越久没有被用户提到、也没有被当前任务调用的对象，在系统里的权重就越低；终将从“活跃候选”中退出，只保留用户曾明确确认过的核心事实。

这样当用户在第 31 轮说“改一下它”时，系统优先绑定的是最近几轮仍在讨论的对象，而不是第 2 轮随口提过、早已无关的文件名。

淡化的边界要清晰：

- **已确认信息**：用户明确说过的，衰减最慢，但仍可被后续明确修正覆盖；
- **推测与候选**：衰减最快，几轮不提就应移出首选范围；
- **澄清历史**：问题本身可以记住“问过、用户拒答”，但不应让旧候选长期占着坑。

淡化不是失忆，而是让状态始终反映“此刻用户在关心什么”，而不是把整段聊天记录压缩成一张越堆越厚的清单。

---

# **5. 第二层：Working State，工作状态**

## **5.1 定位**

Working State 负责管理当前系统正在执行或准备执行的任务。

它回答的问题是：

当前系统正在做什么？做到哪一步？操作对象是什么？

---

## **5.2 管理内容**

Working State 主要包括：

```text
当前任务
当前步骤
当前操作对象
当前候选对象
当前计划
当前中断点
当前可继续动作
```

例如：

```text
当前任务：评估设计文档
当前步骤：已完成目录结构分析，正在分析 retrieval.md
当前操作对象：docs/design/retrieval.md
当前可继续动作：继续分析下一份文档
```

所以当用户说：

继续

系统不应该问：

继续什么？

而应该从 Working State 中判断：

继续分析下一份设计文档。

---

## **5.3 设计原则**

Working State 是处理“继续”“下一步”“按刚才的方式”“改一下它”这类表达的关键。

它不等同于对话状态。

Dialogue State 关注“用户说了什么”，Working State 关注“任务进行到哪”。

---

## **5.4 话题备忘夹：插话、搁置与召回**

真实对话很少单线程。用户可能正在让系统分析 `session.md`，突然插一句“帮我看看下午的日程”；查完日程后，若系统只记住“刚处理了日程”，刚才分析到一半的文档就丢了——这不是好秘书，是金鱼记忆。

Working State 因此需要一层**话题备忘夹**：当用户明显切换主题时，当前任务不是被覆盖，而是被**搁置**进备忘夹，留下：

- 当时在做什么；
- 做到哪一步；
- 下次可以从哪里接着做；
- 一句简短的“待续摘要”，供召回时使用。

备忘夹允许多个搁置项并存，但**活跃工作区始终只有一个**——系统一次只全力推进一件事，其余在备忘夹里安静等着。

搁置触发不必用户明说“先放一下”。只要新输入的意图与当前任务明显无关（查日程、改别的问题、临时闲聊），系统就应自动把当前进度收进备忘夹，转去处理新话题。

**召回**是秘书主动性的体现。侧线任务自然收尾后，系统不应假装无事发生，而应查看备忘夹里是否还有未竟之事，并主动问一句类似：

> 日程已经整理好了。要继续刚才对 session.md 的分析吗？上次停在目录结构那一段。

用户说“继续”即可无缝衔接；说“不用了”则把该项从备忘夹清掉。若备忘夹里有多项，可按搁置时间和用户关注度决定先提哪一项，必要时简短列出候选项。

插话与召回让“继续”不再只依赖同一话题内的 Working State，而是跨话题也能找回被正当打断的工作。

---

# **6. 第三层：Knowledge State，知识状态**

## **6.1 定位**

Knowledge State 负责管理知识检索、知识激活、上下文补充和知识引用。

它回答的问题是：

当前回答需要哪些知识？这些知识是否足够支持回答？

---

## **6.2 管理内容**

Knowledge State 可以包括：

```text
已激活知识单元
候选知识单元
ActivationLink
KnowledgePoint Network
当前检索路径
知识置信度
知识缺口
引用来源
```

例如：

```text
已激活知识单元：
- Trace 设计定位
- Study 与 ActivationLink 的关系
- Working Model 组织思考机制

知识缺口：
- 当前没有关于对话状态管理的设计文档
```

---

## **6.3 设计原则**

Knowledge State 不负责追问用户。

它只负责判断：

```text
知识是否足够？
知识之间是否冲突？
是否需要继续检索？
是否需要提示用户当前依据不足？
```

追问由 Dialogue State 和 Answer State 共同决定。

---

## **6.4 与检索、KPN 和 Working Model 的边界**

Knowledge State 描述的是本轮会话中知识依据的使用状态，而不是完整的知识检索设计。

它关心的是：

```text
当前激活了哪些知识；
哪些知识被采用为回答依据；
当前依据是否充分；
是否暴露知识缺口或冲突；
是否需要提示用户依据边界。
```

但它不定义检索算法、Rerank 细节、KPN 扩展策略或 Working Model 的内部组织过程。这些机制分别由检索、KPN 和临时认知模型相关设计负责。

Session State 只接收这些认知能力的处理结果，并把结果转化为当前会话中的回答策略、澄清策略和状态更新依据。

因此，Knowledge State 不能被实现成一个新的检索总控层。它是会话状态中的知识使用视图，而不是知识大脑的完整知识处理流程。

---

# **7. 第四层：Answer State，回答状态**

## **7.1 定位**

Answer State 负责决定最终系统行为。

它回答的问题是：

当前应该直接回答、追问、给候选项、默认推进，还是拒绝执行？

---

## **7.2 管理内容**

Answer State 主要包括：

```text
回答策略
澄清策略
默认假设
风险等级
可执行性判断
最终输出方式
```

常见回答策略：

```text
DIRECT_ANSWER        直接回答
ASK_CLARIFICATION    追问澄清
OFFER_OPTIONS        给出候选项
SAFE_DEFAULT         采用安全默认值
PARTIAL_ANSWER       给出部分答案
REFUSE_OR_BLOCK      拒绝或阻断
```

---

## **7.3 置信度与风险：何时该问，何时该做**

Answer State 同时持有**把握有多大**（置信度）和**做错了有多糟**（风险）。二者必须一起决策，不能各说各话：高把握不等于可以动手，低把握也不等于一定要追问。

### 行事准则

| 风险 | 把握 | 系统行为 |
| --- | --- | --- |
| 低 | 高 | 直接做，少废话；必要时用一句话交代默认理解即可 |
| 低 | 中 | 采用安全默认继续，并简短说明假设 |
| 低 | 低 | 给候选项或轻量追问 |
| 中 | 高 | 可推进，但点明关键假设，留出口让用户纠正 |
| 中 | 中 | 优先给候选项，而非连环追问 |
| 中 | 低 | 追问或部分回答，不强行执行 |
| 高 | 任意 | **必须确认**——哪怕系统有九成把握，仍要停一步让用户点头 |

“低风险”的典型：改错别字、补充说明、只读查看、可逆的措辞调整。  
“高风险”的典型：删除或覆盖核心文档、批量改动、不可逆操作、对外发布类输出。

同一句话“改一下它”，若绑定的是注释里的笔误，偏低的把握也可以默认推进；若绑定的是 `design.md` 的全文重写，哪怕对象几乎确定，也必须再确认一次。这就是置信度与风险的协同：**把握决定要不要猜，风险决定猜了能不能直接动手。**

用户拒绝澄清时，仍按此矩阵行事，而不是一律“你自己判断就随便做”或一律“不说明白就不动”——拒绝澄清只是关闭追问通道，不关闭在安全默认值下推进低风险的通道，也不关闭在高风险下必须阻断的通道。

---

## **7.4 行动风险与知识依据风险**

Answer State 中的风险不只来自系统即将执行的动作，也来自回答所依赖的知识依据。

行动风险关注的是：

```text
是否会删除、覆盖或批量修改内容；
是否不可逆；
是否会对外发布；
是否会影响用户的重要资产或决策。
```

知识依据风险关注的是：

```text
依据是否当前有效；
是否存在冲突；
是否只是候选理解；
是否需要重新验证；
是否只适合解释历史，而不适合支撑当前回答。
```

因此，即使本轮只是低风险的只读回答，如果知识依据处于待验证、冲突、过期或仅具历史解释价值的状态，系统也不应把结论说成稳定事实。更合适的做法是降低结论强度、说明依据边界，必要时进入查证或冲突检测。

行动风险决定“能不能直接动手”，知识依据风险决定“能不能直接下结论”。两者共同影响 Answer State 的最终策略。

---

# **8. 整体处理流程**

推荐流程如下：

```text
1. 接收用户输入
2. 【优先】是否为系统级打断指令？→ 是则走打断通道（见第 12 节），本轮结束
3. 判断用户意图；若与当前任务无关 → 搁置当前工作进话题备忘夹
4. 更新 Dialogue State（含鲜活度淡化）
5. 尝试从 Dialogue State 和 Working State 解析上下文
6. 判断是否需要知识检索
7. 更新 Knowledge State
8. 判断信息是否充分；结合置信度与风险选择回答策略（见 7.3）
9. 输出回答
10. 更新四层状态；侧线任务若已收尾 → 检查备忘夹并尝试召回
```

关键点是：

每轮对话结束后，都要更新状态，而不是只生成回答。流程不是单行道：打断指令走绿色通道，话题切换走搁置与召回，日常推进走置信度—风险矩阵。

同时，Session State 不替代思维模式选择。

它负责把用户输入整理成当前可处理的对话上下文和工作上下文；至于本轮应该直接回答、快速检索、进入 Working Model、查证还是冲突检测，应由认知路由根据问题类型、风险、证据充分性、复杂度和认知预算等因素决定。

换句话说：

```text
Session State 负责整理“当前用户到底在说什么、要继续什么”；
Cognitive Routing 负责决定“这个问题应该用多深的认知处理路径”；
Retrieval / KPN / Working Model 负责完成被选中的知识处理；
Answer State 负责把处理结果转化为本轮回应。
```

Session 流程是会话控制流程，不是知识处理总流程。

# **9. 信息不足时的处理策略**

## **9.1 可以从上下文补齐**

用户说：

按这个方案继续。

如果 Working State 中已有当前方案，则直接继续。

---

## **9.2 有多个候选对象**

用户说：

修改它。

系统发现候选对象有两个：

```text
design.md
session.md
```

则应该问：

你是指修改 design.md，还是 session.md？

---

## **9.3 缺少关键参数**

用户说：

帮我生成接口设计。

但缺少接口对象。

系统应该问：

这个接口是面向知识检索、文件管理，还是对话状态管理？

---

## **9.4 用户拒绝澄清**

用户说：

你自己判断。

系统不应无限追问，而应参照第 7.3 节的置信度—风险矩阵决定如何推进——拒绝澄清只是关闭追问，不是关闭全部默认策略：

```text
低风险：说明默认假设后继续
中风险：给出候选方案或点明假设后推进
高风险：停止执行并说明缺失信息，不得凭“你自己判断”擅自执行
```

例如：

我先按“知识问答系统内部设计文档”的方向处理，不涉及具体代码实现。

---

# **10. 置信度机制**

系统在解析上下文时，不应只有“成功 / 失败”两个结果，而应有置信度——对“用户到底指的是谁、要什么”的主观把握。

例如：

```text
候选对象：
- design.md：0.82
- session.md：0.76
```

差距很小，在中高风险操作下应追问或给选项；在低风险操作下可结合 7.3 矩阵另作判断。

如果是：

```text
候选对象：
- design.md：0.94
- session.md：0.21
```

对象几乎唯一，但仍要看本轮操作风险：只读引用可直用；删除或重写则仍需确认。

判断依据包括：

```text
最近提及程度（与 Dialogue State 鲜活度一致）
当前任务相关性
用户表达匹配度
对象唯一性
历史确认记录
```

**注意**：置信度只回答“绑定的对象有多准”，不单独决定“能不能执行”。最终动作由第 7.3 节的置信度—风险矩阵裁定；本节负责为矩阵提供“把握”这一维度的输入。

---

# **11. 状态更新原则**

状态更新要遵守几个原则：

## **11.1 新信息优先，但不能随意覆盖旧信息**

如果用户明确修正：

不是 design.md，是 retrieval.md。

则应覆盖当前对象。

如果用户只是补充：

还要看 trace.md。

则应追加候选对象，而不是覆盖。

---

## **11.2 明确表达高于系统推断**

用户明确说的内容优先级最高。

系统推断出来的上下文，只能作为默认假设，不能当成已确认事实。

---

## **11.3 状态需要区分“已确认”和“推测”**

例如：

```text
confirmed_target: design.md
inferred_target: retrieval.md
```

不要混在一起。

---

## **11.4 澄清失败也要进入状态**

如果用户拒绝澄清，系统需要记录：

```text
clarification_status: refused
```

这样下一轮不要重复问同一个问题。

---

## **11.5 定期瘦身：陈旧状态的垃圾回收**

与 4.4 节的淡化相呼应，状态更新还要主动做**垃圾回收**：

- 多轮未被引用、且用户从未确认的推测性对象，从活跃候选中移除；
- 已结束且用户明确表示不再继续的备忘夹条目，删除而非永久挂着；
- 澄清已关闭（用户拒答或已确认）的旧问题，不再参与每轮解析；
- Working State 里已完成且无“继续”预期的步骤，归档到备忘夹或清空，避免“继续”误绑到早已做完的事。

回收的原则是：**宁可少记，不可错绑。** 秘书可以礼貌地问“您指的是哪个？”，不应自信地指向三天前提过、此后再未出现过的文件名。

---

## **11.6 会话状态与长期记忆的边界**

Session State 默认是短期运行状态，不等同于 Trace，也不自动进入长期记忆。

Dialogue State 中的澄清历史、Working State 中的中断点、话题备忘夹里的待续摘要，主要用于维持当前或近期对话的连续性。它们不应因为“发生过”就被当成长期知识沉淀。

只有当本轮处理暴露出对长期记忆有学习价值的信号时，才应提炼为 Learning Event，例如：

```text
知识缺口；
知识冲突；
有效或无效的激活路径；
用户明确纠正；
某类问题反复需要相同处理路径；
Wiki 页面可能需要调整适用边界。
```

普通的追问、拒绝澄清、插话、召回和系统级打断，通常只更新 Session State，不默认触发 Trace 或 Study。

这样可以避免把短期会话管理误做成长期记忆学习，也避免让 Trace 退化成完整对话日志。

---

# **12. 系统级打断：掀桌子通道**

当用户失去耐心，说出“别说了”“停”“清空重来”“算了从头来”这类话时，系统若仍把句子丢进四层模型做指代解析、或继续输出半篇分析、或追问第三个澄清问题，体验会彻底崩坏。

这类输入应走**系统级打断通道**——在所有语义理解之前识别，并**立即**：

1. **停止**当前生成或澄清循环，不再追加输出；
2. **清空**活跃 Working State 与当前 Dialogue 焦点（不必保留“猜一半”的半成品）；
3. **按需重置**话题备忘夹：用户若说“清空重来”，备忘夹一并清空；若只说“停”，可保留备忘夹供之后召回；
4. **回到待命态**：简短确认已停下或已重置，等待下一条明确指令。

打断通道**不**做上下文绑定，**不**追问“您是想停止 A 还是 B”，**不**把“别说了”解释成对某个文档的编辑意见。这是元操作，不是业务对话。

打断与备忘夹的关系：打断是“急刹车”，备忘夹是“临时停车”。用户喊停时系统先刹车；是否保留路边停着的车（搁置任务），由用户措辞区分——“别说了”可能只是停输出，“全部重来”则是连路边一起清空。

---

# **13. 推荐的设计结论**

这个四层模型加上四项进阶能力，核心价值是：

```text
Dialogue State 解决“用户说了什么”——并知道什么该淡、什么该留
Working State 解决“当前做到哪里”——并能在插话后搁置与召回
Knowledge State 解决“依据是什么”
Answer State 解决“现在怎么回应”——按把握与风险一起拍板
话题备忘夹 解决“开小差之后还能接上”
系统级打断 解决“用户随时能喊停、能重来”
```

最终系统不再依赖模型临场发挥，而是形成一套稳定机制：

```text
识别打断 → 理解用户 → 绑定上下文 → 判断缺口
→ 按风险与把握选择策略 → 更新状态 → 必要时召回搁置中的事
```

它从“按流程图走单线问答”升级为**具备人情世故的智能会话状态机**：记得该记的，忘了该忘的，轻重分得清，用户随时能掀桌子。
