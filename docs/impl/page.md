# 第一版测试页面设计

本文档定义第一版用于测试知识大脑设计和实现的网页。

页面目标不是实现完整后台管理系统，也不是做通用知识库产品，而是提供一个现代人工智能知识助手风格的测试控制台，用来验证：

```text
文件导入
  -> 来源状态
  -> 问答
  -> 知识激活
  -> 激活链接查看
  -> 证据引用
  -> 推理过程
  -> 思考记录
  -> 学习候选
```

页面风格可以参考现代对话式人工智能、知识引用、企业知识检索、答案溯源和知识图谱探索产品，但功能边界必须严格落在第一版实际接口和工程设计内。

第一版页面不实现：

```text
网页抓取导入；
即时通讯导入；
正式领域 / 概念创建；
知识图谱手动编辑；
候选自动晋升；
完整生命周期状态流转；
用户权限系统；
多人协作；
提示词暴露和编辑。
```

## 1. 产品定位

第一版页面命名为：

```text
知识大脑控制台
```

它首先是一个面向用户的知识问答工作台，其次才是面向开发和产品验证的测试台。

核心体验是：

```text
中间采用对话式提问；
右侧展示证据和推理过程；
文件导入融入主流程；
知识关系以本次激活路径的形式可视化。
```

页面必须采用双层设计：

```text
默认模式：面向用户，展示回答、来源、知识域、概念、推理摘要和知识缺口；
调试模式：面向开发和测试，展示内部字段、激活链接、思考记录、服务调用和原始响应。
```

页面必须避免把底层实现对象直接暴露为主要产品概念。

例如：

```text
KnowledgeUnit
ActivationLink
PrecompileJob
```

这些只能出现在调试模式、详情抽屉或原始响应中，不能作为默认页面的主要展示对象。

主界面应使用更接近用户理解的表达：

```text
知识
证据
引用
激活路径
缺口
冲突
学习信号
```

展示分层原则：

```text
一级展示：用户语言，例如回答、来源、相关知识域、相关概念；
二级展示：可解释过程，例如问题理解、参考了哪些资料、还有哪些知识缺口；
三级展示：工程调试信息，例如激活链接、服务调用、评分明细、原始响应。
```

## 2. 整体布局

采用三栏沉浸式布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 知识大脑     搜索知识 / 提问...             文件  知识视图  设置 │
├──────────────┬────────────────────────────┬──────────────────┤
│              │                            │                  │
│ 会话 / 记录   │        问答测试区              │   推理与证据区     │
│              │                            │                  │
└──────────────┴────────────────────────────┴──────────────────┘
```

推荐比例：

```text
左侧：22%
中间：53%
右侧：25%
```

右侧解释区默认宽度约 320px，可展开到 480px，也可折叠。

文件管理不作为独立页面切换，而是通过右侧抽屉打开。

知识视图不作为完整知识图谱编辑器，而是展示本次回答相关的知识激活图。

## 3. 顶部导航

顶部导航保持轻量。

内容：

```text
标识：知识大脑
全局输入：搜索知识 / 提问
入口：文件、知识视图、思考记录、设置
```

入口对应能力：

| 入口 | 对应能力 |
| --- | --- |
| 文件 | 来源上传、来源列表、预览、规范化 Markdown |
| 知识视图 | 当前问题 / 思考记录的知识激活图 |
| 思考记录 | 当前思考记录的服务调用、激活、过程记录 |
| 设置 | 接口基础地址、默认认知视角、记录级别、证据要求 |

顶部全局输入可以作为当前对话输入的快捷入口。

## 4. 左侧会话与思考记录区

左侧不只展示对话会话，也要承载第一版最重要的可追踪对象：`trace_id`。

区域名称：

```text
会话 / 思考记录
```

示例：

```text
+ 新问题

今天
────────────────
智能体安全平台如何设计？
模式：推理
记录：trace_xxx

检索增强生成和知识大脑有什么区别？
模式：检索
记录：trace_xxx

昨天
────────────────
企业知识导入测试
模式：检索
记录：trace_xxx
```

支持前端本地能力：

```text
搜索历史；
置顶；
重命名；
删除；
重新加载思考记录。
```

第一版后端没有会话接口时，页面可以用浏览器本地存储保存会话壳。

真实可回放对象必须以 `trace_id` 为准，通过：

```text
GET /api/traces/{traceId}
```

重新加载。

## 5. 中间问答测试区

中间区域是页面核心。

结构：

```text
用户问题
助手回答
证据引用
知识标签
底部输入框
```

### 5.1 问题请求

调用：

```text
POST /api/query
```

请求字段按 `docs/impl/api.md`：

```json
{
  "query": "...",
  "perspective_id": "default",
  "available_context": null,
  "user_intent_hint": null,
  "source_filters": null,
  "outline_filters": null,
  "require_evidence": false,
  "require_external_evidence": false,
  "risk_hint": null,
  "trace_level_hint": null
}
```

底部输入区需要支持：

```text
回车发送；
Shift+Enter 换行；
拖拽文件提问；
要求内部证据开关；
要求外部证据开关；
深度思考开关；
认知视角选择；
思考记录级别选择。
```

`深度思考` 不直接暴露为后端独立模式。

第一版可以映射为：

```text
提高 trace_level_hint；
设置 user_intent_hint；
倾向让路由器选择推理智能体。
```

最终是否使用推理智能体仍由认知路由器决定。

### 5.2 回答展示

回答区展示 `AgentResult`，页面显示时使用中文标签：

```text
回答：answer
模式：mode
使用的知识：used_knowledge
证据：evidence
外部证据候选：external_evidence_candidates
工作模型：working_model
冲突：conflicts
认知缺口：cognitive_gaps
知识缺口：knowledge_gaps
结论引用：claim_refs
推断引用：inference_refs
缺少支持的结论：unsupported_claims
学习信号：learning_signals
模式切换请求：mode_switch_request
思考记录编号：trace_id
```

回答正文采用对话式风格。

引用采用知识引用风格：

```text
根据现有材料，企业智能体安全平台应围绕权限边界、工具调用审计和动态授权设计。[1][2]

来源
[1] 智能体安全规范.md
[2] 企业架构设计.pdf
```

点击引用后：

```text
右侧证据区定位证据；
如果证据带来源信息，允许打开预览或规范化 Markdown；
如果有 trace_id，允许跳转到相关思考记录片段。
```

### 5.3 知识引用卡片

回答下方显示知识引用卡片。

示例：

```text
涉及知识
智能体安全
零信任
接口网关
技能沙箱
```

卡片数据来自以下内部结果，但默认只展示用户可理解的知识域、概念和来源摘要：

```text
使用的知识；
工作模型；
思考记录激活；
证据；
学习信号。
```

第一版不要求后端提供正式知识图谱接口。

如果缺少结构化概念字段，页面可以从 `used_knowledge` 和 `trace` 中降级展示标题、来源和摘要。

## 6. 右侧解释与证据区

右侧区域是本系统区别于普通对话产品的重点，但默认不能像工程日志。

默认模式下，右侧只回答用户关心的三个问题：

```text
这次回答参考了什么；
系统大致如何得到这个回答；
还有哪些不确定或缺失。
```

内部机制放入调试模式。

名称：

```text
解释
```

默认分为三个标签页：

```text
概览
来源
缺口
```

调试模式开启后，增加三个标签页：

```text
激活链接
思考记录
原始响应
```

### 6.1 概览标签页

默认展示用户能理解的推理摘要，不展示服务名、字段名和评分明细。

示例：

```text
回答依据

● 我理解的问题
  企业智能体安全平台如何设计

↓
● 相关知识域
  智能体安全、零信任、接口网关

↓
● 参考来源
  5 条内部来源

↓
● 回答方式
  先整理安全边界，再组织权限、审计和动态授权方案

↓
● 需要注意
  有 1 条结论缺少直接来源支持
```

默认概览可以使用以下用户友好指标：

```text
相关知识域；
相关概念；
引用来源数量；
缺少支持的结论数量；
存在冲突数量；
知识缺口数量；
是否需要外部查证。
```

这些指标由以下内部字段映射得到，但默认不直接显示字段名：

```text
mode；
used_knowledge；
evidence；
working_model；
conflicts；
cognitive_gaps；
knowledge_gaps；
unsupported_claims；
external_evidence_candidates。
```

### 6.2 来源标签页

来源标签页展示用户可理解的证据和文件来源。

每个来源卡片展示：

```text
来源标题；
来源类型；
信任级别；
相关片段；
支持了回答中的哪条结论；
预览 / Markdown 入口。
```

内部证据和外部证据候选必须分开。

外部证据候选展示为：

```text
外部证据候选
候选，不能替代内部来源
```

原因：

```text
external_evidence_candidates 不能替代内部证据；
require_evidence=true 且无 EvidenceRef 时，不得返回确定回答。
```

### 6.3 缺口标签页

缺口标签页展示用户能理解的不足，而不是内部信号。

展示方式：

```text
缺少资料；
证据不足；
存在冲突；
需要外部查证；
系统建议补充的文件或主题。
```

来源字段：

```text
cognitive_gaps；
knowledge_gaps；
conflicts；
unsupported_claims；
learning_signals；
mode_switch_request。
```

默认文案示例：

```text
当前知识库中缺少“动态授权策略”的直接材料。
“技能沙箱”和“接口网关”的边界描述存在重叠，建议补充架构说明。
有 1 条结论缺少直接引用来源。
```

### 6.4 激活链接标签页

激活链接标签页只在调试模式显示。

激活链接是第一版认知结构召回的关键对象。

它表示在某个认知视角下，已有领域 / 概念如何激活知识点 / 知识单元。

页面需要提供专门的激活链接查看入口，但不能把它包装成开放知识图谱关系。

说明文案应使用：

```text
激活链接
```

或：

```text
本次命中的激活路径
```

不要使用：

```text
知识图谱关系
概念关系
实体关系
```

展示内容，页面显示时使用中文标签：

```text
链接编号：link_id；
认知视角：perspective_id；
领域：domain_id / domain_name；
概念：concept_id / concept_name；
知识点：knowledge_point_id；
知识单元：unit_id；
来源：source_id；
状态：status；
置信度：confidence；
激活次数：activation_count；
采用次数：used_count；
正反馈次数：positive_feedback_count；
负反馈次数：negative_feedback_count；
最近激活时间：last_activated_at；
命中路线：hit_route；
评分明细：score_breakdown；
验证支持：validation_support；
支持命中路线：supporting_hit_routes。
```

状态展示规则：

| 状态 | 页面含义 | 展示方式 |
| --- | --- | --- |
| 正式启用 active | 正式认知结构召回路径 | 正常高亮 |
| 候选 candidate | 待验证的学习假设，不参与运行时检索 | 弱高亮，并显示待验证标记 |
| 已停用 disabled | 暂停参与召回 | 灰色 |
| 已丢弃 discarded | 已丢弃，不再参与召回 | 灰色并折叠 |
| 已删除 deleted | 关联对象删除导致失效 | 默认隐藏，可在调试模式显示 |

候选激活链接必须清楚展示验证状态：

```text
candidate 状态的激活链接不进入召回、融合或 used_knowledge；
只使用独立补充检索、EvidenceRef、实际采用和反馈进行验证；
验证通过后晋升 active，验证失败后转 discarded。
```

推荐卡片样式：

```text
智能体安全 -> 技能沙箱
状态：候选 candidate
置信度：0.71
验证：有材料侧召回支持

知识点：kp_xxx
知识单元：unit_xxx
来源：智能体安全规范.md

命中路线
概念匹配 -> 激活链接 -> 材料侧支持
```

点击激活链接卡片后，右侧或弹窗显示：

```text
关联概念；
关联知识点；
关联知识单元；
来源；
本次是否进入 used_knowledge；
是否产生证据引用；
是否产生 learning_signal；
历史激活和采用计数；
状态变化记录。
```

第一版如果没有独立激活链接查询接口，页面从以下数据降级构造，并以中文标签展示：

```text
AgentResult 使用的知识；
AgentResult 证据；
AgentResult 学习信号；
思考记录激活记录；
思考记录服务调用；
检索评分明细。
```

如果思考记录中没有返回激活链接明细，页面必须显示：

```text
本次思考记录未返回激活链接明细。
```

而不是虚构链接。

激活链接标签页的测试目标：

```text
确认 active 状态的激活链接能参与正式召回；
确认 candidate 状态的激活链接只作为弱激活信号；
确认 candidate 状态不会进入 used_knowledge；独立材料与证据只用于由 study 决定晋升 active 或转 discarded；
确认 disabled / discarded / deleted 不参与召回；
确认激活链接命中、采用、弱相关、负反馈能进入思考记录 / 学习信号；
确认激活链接不是开放知识图谱关系。
```

### 6.5 思考记录标签页

思考记录标签页只在调试模式显示。

如果 `GET /api/traces/{traceId}` 可用，展示更细的过程：

```text
问题理解服务；
认知路由器；
检索智能体 / 推理智能体；
检索服务；
证据回链服务；
工作模型服务；
推理服务；
回答组装服务；
学习信号服务；
思考记录服务。
```

页面只展示服务调用和结果摘要，不展示提示词原文。

### 6.6 原始响应标签页

原始响应标签页只在调试模式显示。

展示完整统一响应。

格式：

```json
{
  "ok": true,
  "data": {},
  "warnings": [],
  "error": null,
  "request_id": "...",
  "trace_id": "..."
}
```

失败时展示：

```json
{
  "ok": false,
  "data": null,
  "warnings": [],
  "error": {},
  "request_id": "...",
  "trace_id": "..."
}
```

该标签页用于验证 `docs/impl/runtime.md` 的响应契约。

## 7. 文件抽屉

点击顶部“文件”后打开右侧抽屉。

抽屉宽度：

```text
600px
```

布局：

```text
┌────────────────────────────────────────┐
│ 文件                                  │
├──────────────────┬─────────────────────┤
│ 上传区            │ 来源列表             │
└──────────────────┴─────────────────────┘
```

### 7.1 上传区

调用：

```text
POST /api/sources/files
```

表单字段：

```text
file；
title；
intakePurpose；
trustLevel；
originUri；
producedAt。
```

支持文件类型：

```text
PDF；
DOCX；
MD；
TXT；
HTML。
```

实际可支持类型以来源处理实现为准。

上传进度采用流水线表达，但不能伪造后台进度。

第一版推荐展示：

```text
智能体安全.pdf

✓ 文件上传
● 来源已登记
○ Markdown 转换中 / 等待后台
○ 知识单元构建中 / 等待后台
○ 预编译中 / 等待后台
```

原因：

```text
POST /api/sources/files 不等待知识单元构建 / 预编译完成；
长期候选转换成功后由后台投递知识单元；
temporary_evidence 不进入知识单元。
```

后续状态通过轮询：

```text
GET /api/sources/{sourceId}
```

更新。

### 7.2 来源列表

调用：

```text
GET /api/sources
```

支持过滤：

```text
status；
intakePurpose；
sourceKind。
```

来源卡片示例：

```text
智能体安全.pdf        active
long_term_candidate
信任级别：internal
2026-06-16

预览  Markdown  详情  使用
```

操作：

| 操作 | 接口 |
| --- | --- |
| 预览 | `GET /api/sources/{sourceId}/preview` |
| Markdown | `GET /api/sources/{sourceId}/markdown` |
| 详情 | `GET /api/sources/{sourceId}` |
| 使用 | 写入当前问题请求的 source_filters |

如果第一版 `source_filters` 尚未完整实现，页面仍可保留该测试参数，但必须明确它是问题请求参数，不代表正式文件选择工作流。

## 8. 来源查看器

查看来源时使用弹窗。

尺寸：

```text
80vw
90vh
```

布局：

```text
┌────────────┬────────────────────────────┐
│ 目录 / 搜索 │ 预览 HTML / Markdown 内容    │
└────────────┴────────────────────────────┘
```

能力：

```text
HTML / Markdown 切换；
文本搜索；
复制 sourceId；
查看来源元数据；
从证据点击时自动定位来源；
如果有位置范围信息，高亮相关片段。
```

如果第一版没有精确位置范围定位，页面降级为：

```text
打开来源；
展示证据文本；
提示暂未提供精确位置。
```

## 9. 知识检查器

知识检查器用于检查一次问题请求使用了哪些知识、证据和思考记录。

它不是正式知识管理后台，也不提供领域 / 概念编辑。

入口：

```text
来源卡片；
回答的 used_knowledge；
证据卡片；
思考记录中的 activations；
调试模式中的激活链接卡片；
知识视图节点。
```

布局：

```text
┌──────────────┬────────────────────┬─────────────────┐
│ 来源 / 知识项  │ 详情                │ 关联与使用记录     │
└──────────────┴────────────────────┴─────────────────┘
```

左侧：

```text
来源；
使用的知识；
激活链接；
证据；
学习信号；
缺口；
冲突。
```

中间：

```text
内容；
摘要；
来源；
结论引用；
推断引用。
```

右侧：

```text
相关概念；
激活链接状态；
激活链接命中路线；
知识缺口；
认知缺口；
冲突；
思考记录激活；
被哪些回答采用。
```

数据来源：

```text
POST /api/query 的 AgentResult；
GET /api/traces/{traceId}；
GET /api/sources/{sourceId}；
GET /api/sources/{sourceId}/preview；
GET /api/sources/{sourceId}/markdown。
```

第一版不要求新增知识单元查询接口。

第一版也不强制新增独立激活链接查询接口。

但只要 `TraceRecord.activations`、`used_knowledge` 或检索结果中携带链接编号或链接摘要，知识检查器必须能展示激活链接详情。

激活链接详情至少包括：

```text
该链接所属认知视角；
从哪个领域 / 概念出发；
激活了哪个知识点 / 知识单元；
当前状态；
本次是否命中；
本次是否被采用；
是否有证据支持；
是否产生学习信号；
历史使用和反馈计数；
状态变化原因。
```

如果只拿到链接编号，没有详情，页面应展示链接编号和缺失字段提示。

## 10. 知识视图

顶部“知识视图”打开知识页面或知识视图抽屉。

第一版知识视图默认不做全屏图谱，而采用知识域浏览布局：

```text
┌──────────────────┬────────────────────────────────────────────┐
│ 知识域列表         │ 知识域详情                                  │
│                  │                                            │
│ 安全              │ 名称、描述、相关概念、相关文件、最近使用        │
│ 架构              │                                            │
│ 智能体            │ 高级信息                                    │
│ 金融              │ 激活链接、思考记录、本次激活图                  │
└──────────────────┴────────────────────────────────────────────┘
```

第一版没有正式知识域管理和知识域查询 API 时，该页面必须降级为“当前问题 / 当前思考记录相关的知识域视图”。

数据来源只能来自：

```text
POST /api/query 返回的 AgentResult；
GET /api/traces/{traceId} 返回的思考记录；
GET /api/study/candidates 返回的学习候选；
GET /api/sources 和 source 详情。
```

页面不得假设已经存在：

```text
GET /api/domains；
GET /api/domains/{domainId}；
GET /api/domains/{domainId}/concepts；
GET /api/domains/{domainId}/activation-links。
```

如果后端后续增加只读知识域查询 API，页面可以从降级视图切换为全量知识域视图。

左侧是知识域列表。

支持：

```text
搜索知识域；
按最近使用排序；
按知识数量排序；
显示知识域下概念数量；
显示相关文件数量；
显示最近被问到的问题数量。
```

上述统计在第一版只能基于当前 trace、当前会话历史和已返回结果计算；不能展示为全局精确统计。

右侧默认展示知识域详情。

内容：

```text
知识域名称；
知识域说明；
所属认知视角；
相关概念；
相关文件；
最近被问到的问题；
最近被引用的回答；
系统识别到的知识缺口；
建议补充的材料。
```

如果缺少全量知识域数据，右侧标题应显示：

```text
本次相关知识域
```

而不是暗示这是完整知识域档案。

右侧详情页面包含三个标签页：

```text
概览；
相关知识；
高级信息。
```

### 10.1 概览标签页

概览标签页面向用户。

展示：

```text
这个知识域是什么；
包含哪些核心概念；
来自哪些文件；
最近在哪些问题中被使用；
当前还缺哪些知识。
```

### 10.2 相关知识标签页

相关知识标签页展示用户可理解的知识条目，不直接暴露知识单元和知识点编号。

展示：

```text
知识标题；
摘要；
来源文件；
相关概念；
最近使用；
证据片段；
查看来源入口。
```

如果后端只返回 `used_knowledge` 或思考记录激活信息，页面可以降级为“本次问题相关知识”。

### 10.3 高级信息标签页

高级信息标签页默认折叠，只有打开调试模式后显示。

它包含：

```text
激活链接列表；
本次激活图；
思考记录；
原始对象摘要。
```

激活链接列表用于查看当前知识域下的激活路径。

展示：

```text
概念；
知识点；
知识单元；
来源；
状态；
置信度；
激活次数；
采用次数；
证据支持；
最近激活时间。
```

普通用户不需要看到“激活链接”这个词。

默认模式下可以把它包装成：

```text
这个知识域会关联到哪些知识
```

调试模式下才显示：

```text
ActivationLink；
active / candidate / disabled / discarded / deleted；
score_breakdown；
validation_support；
supporting_hit_routes。
```

### 10.4 本次激活图

本次激活图是高级信息的一部分，不作为知识页面默认主视图。

它的含义必须是：

```text
本次回答的知识激活图
```

而不是全局完整知识图谱。

节点类型：

| 类型 | 含义 | 颜色 |
| --- | --- | --- |
| 问题 | 用户问题 | 蓝色 |
| 概念 | 概念或主题 | 绿色 |
| 知识 | 被采用知识 | 紫色 |
| 激活链接 | 激活链接 | 青色 |
| 证据 | 来源证据 | 黄色 |
| 缺口 | 缺口 | 红色 |
| 冲突 | 冲突 | 橙色 |
| 思考记录 | 过程记录 | 灰色 |

边类型：

```text
问题 -> 概念；
概念 -> 激活链接；
激活链接 -> 知识；
知识 -> 证据；
回答 -> 结论；
结论 -> 证据；
思考记录 -> 服务调用；
思考记录 -> 激活记录；
缺口 -> 学习信号。
```

数据来源：

```text
使用的知识；
证据；
工作模型；
激活记录；
激活链接；
认知缺口；
知识缺口；
冲突；
学习信号；
思考记录；
服务调用。
```

点击节点后右侧显示详情。

如果字段不足，页面应降级展示原始对象摘要，不虚构不存在的关系。

激活链接节点只表示认知激活路径，不表示两个概念之间的知识图谱关系。

图上推荐表达为：

```text
概念
  -> 激活链接
  -> 知识点
  -> 知识单元
  -> 证据
```

不要表达为：

```text
概念
  -> 概念
```

除非思考记录或后端显式返回这种关系。

## 11. 激活路径可视化

激活路径是高级解释和调试模式的重点视觉元素。

默认模式不展示完整激活路径，只展示用户可理解的摘要：

```text
本次回答主要参考了“智能体安全”“零信任”“接口网关”三个知识方向。
```

用户点击“查看过程”或开启调试模式后，才展示更细的激活路径。

高级模式路径：

```text
用户问题
  ↓
问题理解
  ↓
认知路由
  ↓
检索智能体 / 推理智能体
  ↓
知识激活
  ↓
证据回链
  ↓
回答组装
```

如果思考记录返回 activations，可以进一步展示：

```text
智能体安全
  ↓
技能沙箱
  ↓
接口网关
  ↓
令牌服务
```

文案必须使用：

```text
本次激活路径
```

不要写成：

```text
完整知识图谱路径
```

## 12. 学习候选

学习候选不作为主页面。

它出现在：

```text

知识检查器；
思考记录详情；
设置或调试抽屉。
```

调用：

```text
GET /api/study/candidates
```

展示：

```text
学习候选；
来源思考记录；
相关缺口；
相关证据；
候选原因；
创建时间。
```

第一版只展示候选，不执行自动晋升。

页面必须避免提供：

```text
一键加入长期记忆；
自动创建领域；
自动创建概念；
自动修改知识结构。
```

```text
必要数据目录可写；

request_id；
警告；
错误。
```

页面还应方便测试错误场景：


```text

无需 npm / Vite 等前端构建链；
```

约束：

```text

所有接口响应必须能切换查看原始响应；
所有错误必须展示 request_id；
trace_id 必须可复制；
```

## 17. 第一版验收标准

页面完成后，应能验证以下链路：

```text
1. 健康检查可查看；
2. 文件可以上传为来源；
3. 来源列表可以过滤；
4. 来源详情可以查看；
5. 预览 HTML 和 Markdown 可以打开；
6. 用户可以发起问题请求；
7. 回答可以展示；
8. 证据可以点击；
9. require_evidence 行为可以测试；
10. 默认模式不直接暴露激活链接、服务调用、评分明细和原始响应；
11. 解释面板可以展示用户可理解的回答依据、来源和缺口；
12. 知识页面左侧可以浏览知识域；
13. 知识页面右侧可以查看知识域详情和相关知识；
14. 调试模式可以查看激活链接；
15. 激活链接可以从知识检查器查看详情；
16. active / candidate / disabled / discarded / deleted 的展示语义正确；
17. candidate 状态激活链接的验证支持状态可以查看；
18. trace_id 可以加载思考记录；
19. 缺口 / 冲突 / 缺少支持的结论可以展示；
20. 学习候选可以查看；
21. 原始响应可以在调试模式查看；
22. 所有错误都符合统一响应契约。
```

## 18. 总结

第一版测试页面应是一个现代人工智能知识问答工作台。

它的主体验是：

```text
中间提问；
右侧解释；
文件以内嵌抽屉进入；
知识页面以知识域浏览为主；
激活链接、思考记录和原始响应只在调试模式用于工程验证。
```

页面可以呈现现代对话、知识引用、企业检索、过程透明和图谱探索的体验特征，但不能超出第一版系统实际能力。
